# Tauranga SME — gated assets

This folder serves the **per-purchase unguessable URLs** generated by
`server/lib/products.ts → buildAccessLinks()` and emailed to buyers in
`taurangaAccessEmail()` (see `server/lib/email.ts`).

The slugs are random `base64url` strings produced at webhook time. There is
**no DB lookup** — buyers receive the link in their access email after
Stripe `checkout.session.completed` fires.

## Expected layout

For v1 soft-launch, the simplest pattern is to drop the same template files
into a single canonical folder (e.g. `_v1/`) and either:

1. **Symlink** the random slug folders to the canonical folder before launch, or
2. Switch `buildAccessLinks()` to a fixed slug per asset (less unguessable but
   simpler) until volume warrants per-purchase tokens.

For v1 we recommend option 2: replace the `randomBytes(24)` calls with a single
shared slug per asset — `pia/`, `ai-use-policy/`, `tool-register/`,
`te-tiriti-checklist/`, `four-session-walkthrough/` — and rely on Stripe
receipts as the access control. Bump to per-purchase tokens once volume merits.

## Asset checklist

- `pia/nz-privacy-impact-assessment-template.pdf` — 1-page PIA template (Silver+)
- `ai-use-policy/internal-ai-use-policy-template.pdf` — 1-page Use Policy (Silver+)
- `tool-register/ai-tool-register-template.xlsx` — Tool Register sheet (Silver+)
- `te-tiriti-checklist/te-tiriti-maori-data-sovereignty-checklist.pdf` — Gold-only
- `four-session-walkthrough/index.html` — Video walkthrough page (all tiers)

For pre-launch, a "recording in progress" placeholder MP4 is fine — the funnel
can sell before recordings finish.

This file (and the folder structure) lives in the repo so deploys ship the
right shape; populate the actual files before flipping Stripe to live mode.
